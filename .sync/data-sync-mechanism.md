## 🚨 同步铁律（不可违反）

### 1. 数据源唯一性原则
- **唯一数据源**：`portfolio/holdings.md`
- **禁止**：任何端点维护独立持仓数据
- **强制**：所有分析必须基于最新读取的 holdings.md

### 2. 同步触发条件

| 触发事件 | 执行动作 | 责任人 |
|---------|---------|--------|
| 用户发送持仓到任一端 | 立即读取 holdings.md | 两端 AI |
| 用户要求分析持仓 | 先读取 holdings.md，再分析 | 两端 AI |
| 每次对话开始 | 验证 holdings.md 时间戳 | 两端 AI |
| 用户指出数据错误 | 立即重新读取 + 致歉 + 修正 | 两端 AI |

### 3. 时间戳验证机制

**每次读取 holdings.md 时必须验证**：
```
最后更新时间 → 是否在 1 小时内？
    ↓ 是 → 使用数据
    ↓ 否 → 询问用户"持仓数据是否需要更新？"
```

## 📋 同步检查清单

### 每次分析前必须确认（强制！）

- [ ] **1. 读取最新 holdings.md**（不是缓存！）

- ❌ 说"大约"、"可能"、"应该"等模糊词汇
- ❌ 不读取文件就分析持仓

## 🔄 同步流程

### 用户发送持仓到电脑端

```
用户发送 → 电脑端接收 → 更新 holdings.md
    ↓
微信端收到通知 → 立即读取 holdings.md → 验证时间戳
    ↓
确认同步完成 → 回复用户"数据已同步"
```

### 用户要求分析持仓

```
用户请求 → AI 读取 holdings.md → 验证时间戳
    ↓
时间戳新鲜（<1 小时）→ 基于数据分析
    ↓
时间戳过期（>1 小时）→ 询问用户"是否需要先更新数据？"
```

## 📊 数据验证规则

### 自动验证（每次读取必须执行）

| 验证项 | 计算方法 | 容差 |
|--------|---------|------|
| **持仓总数** | 逐行计数 A 股 + 港股 | 0 |
| **A 股市值** | 所有 A 股市值求和 | ±0.01 万 |
| **港股市值** | 所有港股市值求和 | ±0.01 万 |
| **总市值** | A 股市值 + 港股市值 | ±0.01 万 |
| **浮动盈亏** | 逐只股票浮盈求和 | ±0.01 万 |

### 验证失败处理

```
验证失败 → 立即报告用户 → 暂停分析 → 等待修正
```

## ⚠️ 错误处理机制

### 发现数据不一致时

| 问题 | 处理流程 |
|------|---------|
| 持仓数量对不上 | 立即重新读取 + 逐行计数 + 报告用户 |
| 市值计算错误 | 重新计算 + 报告差异 + 询问用户 |
| 时间戳过期 | 询问用户"是否需要先更新数据？" |
| 用户指出错误 | 立即致歉 + 重新读取 + 修正分析 |

### 错误记录

所有数据错误必须记录到：`memory/data-sync-errors.md`
